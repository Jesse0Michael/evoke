## Overview

The registry is the distribution backend for Evoke files — think **"Docker Hub for `.evoke` files."** It stores immutable `.evoke` artifact versions and the accounts that publish them, and serves push / pull / list over REST/HTTP.

Like the format itself, the registry does not require files to declare a category or type. A registry entry that humans call "a character" or "an outfit" is, to the CLI, just an Evoke document.

## References

Registry references use the `@namespace/name` format on the CLI:

```console
$ evoke generate @jesse/sumi @jesse/winter-coat
```

The CLI downloads the latest version, caches it in `~/.evoke/library/`, and tracks the SHA256 in a manifest file for reproducibility.

## Architecture

- **REST/HTTP** with spec-first design: `api/openapi.yaml` is the source of truth.
- **ent + Postgres** for persistence. Schemas: `User`, `Identity`, `Artifact`, `Version`.
- **Google OIDC auth** — the CLI obtains a Google ID token via browser loopback + PKCE, exchanges it at the registry, and receives the registry's own JWT pair.
- **Content in Postgres** — `.evoke` files are tiny; content is stored in a `Version.content` column, addressed by SHA256. The `store.Store` interface allows swapping to S3/MinIO later.
- **Generated Go client** in `internal/client/client.gen.go` — generated by oapi-codegen from the OpenAPI spec.

## Endpoints

```text
GET    /health
POST   /v1/tokens/google                 exchange a Google ID token → {user, access_token, refresh_token}
GET    /v1/account            [auth]     the authenticated account ("me")
PATCH  /v1/account            [auth]     update username
DELETE /v1/account            [auth]     delete account (cascades)
PUT    /v1/{namespace}/{name} [auth]     push raw .evoke body → new immutable version
GET    /v1/{namespace}/{name}            list versions
GET    /v1/{namespace}/{name}/{version}  pull raw .evoke bytes (X-Evoke-SHA256 header)
```

## Out of scope

- **Workflow topology.** Checkpoints, samplers, detailer passes, upscaling, ControlNet, IP-Adapter — this is a graph-and-execution problem, not a character-data merge problem. Workflow support stays an extension point.
- **Runtime agent memory.** Session history, memory extraction, vector retrieval, and per-user long-term memory belong to an external runtime. Evoke focuses on authored, versioned source data.
